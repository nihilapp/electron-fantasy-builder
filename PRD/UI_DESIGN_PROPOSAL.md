# UI 제안서 — Fantasy Builder

**목적**: 본 문서는 Fantasy Builder의 **UI 제안서 작성 전 분석 및 가이드**입니다.  
제품 특성, 사용 시나리오, 필수 UI 패턴을 정리하여 **구체적인 디자인(화면·컴포넌트·인터랙션)**을 설계할 때 참고할 수 있도록 합니다.

**관련 문서**: [PRD.md](./PRD.md)

---

## 1. 제품·사용자 요약 (디자인 관점)

### 1.1. 제품이 하는 일

- **World Building Tool**: 작품/세계관 설정을 **프로젝트 단위**로 정리·기록하는 **데스크톱 앱**(Electron).
- **핵심 개념**:  
  - **프로젝트** = 하나의 작품/세계관.  
  - **특성(Traits)** / **어빌리티(Abilities)** = 전역 풀 + 프로젝트 전용 풀.  
  - **설정 엔티티** = 인물, 종족, 아이템, 지역, 국가, 조직, 사건, 전승 등.  
  - **매칭** = 엔티티에 특성·어빌리티를 붙여 “이 캐릭터/종족의 성향·능력”을 구조화.

### 1.2. 타깃 사용자 (UI 톤·복잡도 결정)

| 대상 | 사용 맥락 | UI 시사점 |
|------|------------|-----------|
| 작가, 시나리오 작가 | 인물·조직·국가 관계, 배경 설정 정리 | 텍스트(이름, 설명, 로그라인) 비중 큼. 긴 글 입력·표시에 유리한 레이아웃 필요. |
| 게임 기획자 | 캐릭터 스탯, 종족/직업 특성, 스킬·어빌리티 체계 | 특성·어빌리티 **선택/매칭** UI가 핵심. 분류·검색·다중 선택이 자주 쓰임. |
| 만화가·기타 창작자 | 등장인물, 세계관 일관성 유지 | “누가 어떤 특성을 가지는지” 한눈에 보이는 뷰가 중요. |

- **공통**: 오프라인에서 **집중해서 정리·기록**하는 도구. 화면이 산만하지 않고, **정보 밀도와 가독성**을 동시에 만족하는 디자인이 적합합니다.

### 1.3. 기술·플랫폼 제약

- **Electron 데스크톱**: 웹과 유사하지만 창 크기·해상도 제어 가능. 모바일 퍼스트는 아니며, **사이드바 + 메인** 구조가 자연스러움.
- **스택**: Vue 3 (Composition API), Tailwind CSS 4, Pinia.  
  → UI 제안 시 **Tailwind 기반** 컴포넌트·유틸리티 클래스, Vue 단일 파일 컴포넌트 구조를 전제로 하면 구현 연계가 쉽습니다.
- **현재 레이아웃**: 커스텀 타이틀바, 좌측 사이드바(또는 프로젝트별 카테고리 사이드바), 메인 영역.  
  → 제안서에서 **새 화면/섹션**을 추가할 때 이 구조를 유지할지, 예외(풀스크린 모달 등)를 둘지 명시하면 좋습니다.

## 2. UI에서 다뤄야 할 데이터·행위 (특징 정리)

아래는 **디자인 시 반드시 고려해야 할** 데이터 유형과 사용자 행위입니다.  
텍스트 입력, 특성/어빌리티 선택, 목록·상세, 관계 등이 반복되므로 **공통 컴포넌트·패턴**을 정의하기 좋습니다.

### 2.1. 텍스트 입력이 많은 곳

- **짧은 텍스트 (한 줄)**  
  - 프로젝트명, 인물명, 특성명, 어빌리티명, 장르, 별칭 등.  
  - **필수/선택** 구분, placeholder, 검증 에러 메시지 노출 위치를 디자인에 포함할 것.
- **중·장문 텍스트**  
  - 프로젝트 설명, 인물 로그라인, 특성 설명, 코어 규칙 내용 등.  
  - **textarea** 또는 에디터 형태. 스크롤, 최소 높이, 선택 시 포커스 스타일을 정해 두면 일관됩니다.
- **검색/필터**  
  - 목록 화면의 `searchKeyword`, `searchType`(어떤 필드로 검색할지).  
  - 입력창 + 검색 타입 선택(드롭다운 또는 탭) 조합을 하나의 **검색 블록** 패턴으로 두기 좋습니다.

**제안서에 넣을 내용 예시**  
- 단일 줄 입력: 라벨 위치(위/좌측), 필수 표시(*), 에러 시 테두리/메시지 색.  
- 여러 줄 입력: 높이, 패딩, 폼 내 다른 필드와의 간격.  
- 검색: 키워드 입력 + 검색 타입 선택 레이아웃, 버튼 유무(엔터만 vs 검색 버튼).

### 2.2. 특성(Trait) / 어빌리티(Ability) 선택·매칭

- **풀 구조**  
  - **전역 풀**: 모든 프로젝트에서 공통 사용.  
  - **프로젝트 전용 풀**: 해당 작품에서만 쓰는 특성/어빌리티.  
  - UI에서 “전역 / 프로젝트 / 통합 검색” 구분이 명확해야 함.
- **선택 방식**  
  - 엔티티(인물, 종족 등) **상세/편집** 화면에서 “이 엔티티에 붙일 특성·어빌리티”를 **다중 선택**.  
  - 전역 + 프로젝트 풀을 **검색(키워드, 대분류·중분류, 적용 대상)** 후 목록에서 선택.  
  - 이미 매칭된 항목은 **태그/칩** 형태로 표시하고, 제거 버튼 제공.
- **표시 정보**  
  - 특성: 이름, 설명(툴팁 또는 접기/펼치기), 대분류·중분류, 적용 대상, 상충 특성(있을 경우).  
  - 어빌리티: 이름, 타입, 대분류, 설명, 대상/대미지 타입 등.  
  - 목록이 길어지므로 **페이징·가상 스크롤** 중 하나를 전략으로 정하고, 카드/행 높이를 일정하게 두면 좋습니다.

**제안서에 넣을 내용 예시**  
- “특성 선택기” 1개 화면/컴포넌트: 검색 영역 + (전역/프로젝트 탭 또는 필터) + 선택 가능 목록 + 이미 선택된 항목(칩) 영역.  
- “어빌리티 선택기”는 특성과 필드만 다르고 **같은 패턴**으로 통일할지 여부.  
- 모달로 띄울지, 메인 영역 인라인으로 넣을지.

### 2.3. 목록·상세·CRUD 공통 패턴

- **엔티티 종류**  
  - 코어 설정, 종족/생물, 인물, 아이템, 지역, 국가, 단체, 사건, 전승 등.  
  - 모두 **프로젝트(prjNo) 스코프** 하에 목록·상세·생성·수정·삭제(소프트 삭제)를 제공합니다.
- **목록 화면**  
  - **페이징** (page, pageSize), **검색** (searchKeyword, searchType).  
  - 행: 최소 식별용(이름 + 선택 시 상세로 이동), 필요 시 부가 정보 1~2열.  
  - “추가” 버튼 위치(상단 우측 등) 통일.
- **상세/편집 화면**  
  - 필드 배치: 식별자(이름 등) 상단, 그 다음 메타/분류, 그 다음 **특성·어빌리티 매칭** 영역, 마지막에 관계(향후) 등.  
  - 저장/취소 버튼 위치, 로딩·에러 메시지 영역.

**제안서에 넣을 내용 예시**  
- 목록: 테이블 vs 카드 vs 리스트 중 프로젝트 기본 스타일 1가지.  
- 상세: 단일 컬럼 vs 2단(요약 좌측 + 상세 우측) 등 레이아웃 원칙.  
- “특성/능력 관리” 전용 화면: 전역·프로젝트 탭, 목록 + 검색 + CRUD 폼 배치.

### 2.4. 관계(인물–인물, 인물–조직 등)

- PRD상 **엔티티 간 관계**는 향후 기능이지만, UI 제안서 단계에서 **자리만** 잡아 두면 좋습니다.  
- 예: 상세 화면 하단 “관계” 섹션, “대상 엔티티 + 관계 유형” 선택기, 관계 목록 표시 방식(리스트/그래프 등).

## 3. 디자인 방향 제안 (제안서 작성 시 채울 항목)

아래는 **구체적인 UI 제안서**를 쓸 때 **결정해야 할 항목**을 정리한 것입니다.  
각 항목에 대해 “선택한 방향 + 이유”를 문서에 적으면, 이후 디자인·구현이 일관되게 진행됩니다.

### 3.1. 톤 & 비주얼

- **분위기**: 창작 도구답게 **차분하고, 정리된 느낌**을 유지할지, 아니면 장르(판타지 등)를 살린 **테마 컬러/일러스트**를 넣을지.
- **타이포그래피**: 제목/본문/캡션 계층, 장문 읽기 편한 줄 간격·최대 너비.
- **색·대비**: 필수 입력 강조, 에러, 비활성, “선택됨” 상태 색을 PRD/디자인 시스템과 맞출 것.

### 3.2. 레이아웃 원칙

- **앱 전체**: 타이틀바 + (선택) 앱 레벨 사이드바 + **라우트별 메인 영역**.  
  - 프로젝트 상세는 **프로젝트 전용 사이드바(카테고리)** + 메인. (현재 구조 유지 여부.)
- **메인 영역**:  
  - 목록은 **전체 너비** vs **최대 너비 고정 + 중앙 정렬**.  
  - 폼/상세는 **카드 하나에 담을지**, **여러 섹션 카드**로 나눌지.
- **반응형**: Electron이지만 창 크기 변동을 고려할지.  
  - 예: 사이드바 260px 고정 vs 좁을 때 접기/드로어.

### 3.3. 공통 컴포넌트 (제안서에서 정의 권장)

| 용도 | 고려 사항 |
|------|-----------|
| **단일 줄 입력** | 라벨, placeholder, 필수 표시, 에러 메시지, 비활성 상태. |
| **여러 줄 입력** | 최소/최대 높이, 리사이즈 가능 여부. |
| **검색 블록** | 키워드 입력 + 검색 타입(필드) 선택 + (선택) 검색 버튼. |
| **특성/어빌리티 선택기** | 검색, 전역/프로젝트 구분, 다중 선택, 선택된 항목 칩, 페이징/스크롤. |
| **목록(테이블/리스트)** | 헤더, 행 클릭 시 상세 이동, 빈 상태, 로딩, 페이징 UI. |
| **카드(상세 요약)** | 제목, 2~3개 핵심 필드, “관리/편집” 링크·버튼. |
| **모달/드로어** | 선택기·확인 다이얼로그 등. ESC·오버레이 클릭 동작. |

### 3.4. 프로젝트 상세 내 카테고리별 화면

PRD·TODO에 따른 **설정 카테고리**별로, 다음을 제안서에 적으면 됩니다.

- **프로젝트 개요**: 어떤 필드를 어떤 배치로 보여 줄지 (요약 카드 vs 인라인 편집).
- **특성/능력 관리**: 전역·프로젝트 탭, 목록 + 검색 + CRUD, (선택) 상세 슬라이드/패널.
- **코어 설정, 종족, 인물, …**: 각각 목록 + 상세/폼 구조, 인물/종족은 **특성·어빌리티 매칭** 영역 포함.
- **관계**: 자리만 잡을지, 간단한 “대상 + 유형” 목록 UI까지 제안할지.

## 4. 제안서 작성 체크리스트

UI 제안서 초안을 쓸 때 아래를 채우면, 개발 연계가 수월합니다.

- [ ] **타깃 사용자·시나리오** 한 줄 요약 (본 문서 §1 참고).
- [ ] **텍스트 입력** 규칙: 단일 줄/여러 줄 기본 스타일, 필수 표시, 에러 표시.
- [ ] **특성/어빌리티 선택** UI: 검색 + 풀 구분 + 다중 선택 + 선택 결과 표시 방식.
- [ ] **목록** 공통: 테이블/리스트/카드 중 기본, 페이징·검색 UI 위치.
- [ ] **상세/폼** 공통: 섹션 구분, 저장/취소, 매칭 영역 배치.
- [ ] **레이아웃**: 사이드바 너비, 메인 최대 너비, 반응(접기 등) 여부.
- [ ] **색·타이포·아이콘**: 기존 Tailwind·아이콘 세트와의 정합성.
- [ ] **(선택)** 관계 UI: 영역만 둘지, 목록/선택기 스케치까지 할지.

---

## 5. 참고: 현재 구현 상태

- **구현됨**: 메인/프로젝트 목록/프로젝트 생성/프로젝트 상세(카테고리 사이드바 + 기본 정보 등), 로딩 화면, Tailwind·VueIcon 기반 스타일.
- **미구현·확장 예정**: 프로젝트 상세 내 카테고리별 메인 뷰(특성·어빌리티, 코어 설정, 인물, …), 특성/어빌리티 선택기, 관계 UI.

제안서는 **미구현 영역**을 중심으로 “어떤 화면에 어떤 컴포넌트를 어떻게 배치할지”를 구체화하는 데 쓰면 됩니다.
