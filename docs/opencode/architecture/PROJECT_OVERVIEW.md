## Fantasy Builder란 무엇인가요?

Fantasy Builder는 작가, 게임 기획자, 만화가, 시나리오 작가 등 창작 전문가를 위해 설계된 **데스크톱 세계관 구축 도구**입니다. 이 도구는 가상의 세계, 캐릭터, 배경 설정을 프로젝트 기반 구조로 정리하고 문서화하는 체계적인 접근 방식을 제공합니다.

### 핵심 목적
- **체계적인 세계관 구축**: 가상의 배경 설정, 캐릭터, 이야기(Lore)를 구조적으로 정리
- **프로젝트 기반 관리**: 각 창작물(소설, 게임, 만화 등)마다 독립적인 프로젝트 공간 제공
- **캐릭터 & 엔티티 관리**: 캐릭터, 크리처, 조직, 장소 간의 특성, 능력, 관계 관리
- **오프라인 우선 접근**: 인터넷 없이 주요 기능 작동 (선택적 온라인 기능 제공)

### 타겟 사용자
| 사용자 유형 | 주요 사용 사례 |
|-----------|------------------|
| **작가** | 소설 캐릭터 프로필, 세계관 구축, 줄거리 구성 |
| **게임 기획자** | 캐릭터 스탯, 종족 특성, 스킬 시스템, 팩션(세력) 관계 |
| **만화가** | 캐릭터 디자인, 배경 설정, 스토리 타임라인 |
| **시나리오 작가** | 캐릭터 관계도, 장소 세부 정보, 사건 시퀀스 |

## 주요 기능 (Key Features)

### 1. 프로젝트 관리
- **독립적인 프로젝트 공간**: 각 작품마다 격리된 환경 제공
- **프로젝트 메타데이터**: 장르, 설명, 커버 이미지, 버전 추적
- **빠른 탐색**: 프로젝트 간 빠른 전환

### 2. 특성 & 능력 시스템 (Traits & Abilities System)
- **전역 풀 (Global Pool)**: 여러 프로젝트에서 재사용 가능한 특성 및 능력
- **프로젝트 전용**: 각 프로젝트에 고유한 커스텀 특성 및 능력
- **범주화**: 유형, 적용 대상, 관계별로 정리
- **충돌 해결**: 상충되는 특성 및 능력 처리

### 3. 엔티티 관리 (Entity Management)
- **캐릭터**: 특성, 능력, 관계를 포함한 상세 캐릭터 프로필
- **크리처/종족**: 종족 및 크리처 관리
- **조직**: 팩션, 길드, 회사, 집단
- **장소**: 지역, 국가, 특정 장소
- **아이템**: 무기, 유물, 중요한 사물
- **이벤트**: 역사적 사건 및 타임라인 관리
- **로어 (Lore)**: 신화, 전설, 배경 이야기

### 4. 관계 시스템 (Relationship System)
- **엔티티 연결**: 캐릭터와 조직 연결, 장소와 국가 연결 등
- **시각적 탐색**: 상호 연결된 엔티티를 통해 관계 탐색
- **관계 유형**: 다양한 연결 종류 정의 (가족, 고용, 동맹 등)

## 기술 아키텍처 개요 (Technical Architecture Overview)

### 애플리케이션 유형
- **Electron 데스크톱 애플리케이션**: 크로스 플랫폼 (Windows, macOS, Linux)
- **하이브리드 아키텍처**: 메인 프로세스 (Node.js) + 렌더러 프로세스 (Vue.js)
- **내부 API**: 메인-렌더러 통신을 위한 Hono 기반 HTTP 서버

### 기술 스택
- **Frontend**: Vue 3 (Composition API), TypeScript, Tailwind CSS v4
- **Backend**: Hono (HTTP API), Drizzle ORM
- **Database**: SQLite (로컬) 및 PostgreSQL 지원 (원격)
- **Build System**: electron-vite, pnpm 패키지 매니저
- **State Management**: Pinia 스토어
- **Validation**: 런타임 타입 체크를 위한 Zod 스키마

### 데이터 아키텍처
- **이중 데이터베이스 지원**: 오프라인용 로컬 SQLite, 온라인 기능용 원격 PostgreSQL
- **스키마 주도**: TypeScript 우선 데이터베이스 설계를 갖춘 Drizzle ORM
- **소프트 삭제 (Soft Deletes)**: 삭제 플래그를 사용한 데이터 보존
- **감사 추적 (Audit Trail)**: 모든 엔티티에 대한 생성/수정 추적

## 개발 철학 (Development Philosophy)

### 설계 원칙
1. **오프라인 우선**: 인터넷 의존성 없는 핵심 기능
2. **데이터 주권**: 사용자가 자신의 데이터를 소유하고 내보낼 수 있음
3. **모듈식 아키텍처**: 독립적이고 유지보수 가능한 컴포넌트
4. **타입 안전성**: 런타임 검증을 포함한 완전한 TypeScript 커버리지
5. **사용자 경험**: 최소한의 학습 곡선과 직관적인 인터페이스

### 코드 구성
- **계층형 아키텍처**: Controller → Service → Database 패턴
- **타입 우선 개발**: Zod 스키마로 데이터 계약 정의
- **컴포넌트 기반 UI**: 일관된 스타일의 재사용 가능한 Vue 컴포넌트
- **API 우선 설계**: UI 구현 전 백엔드 API 설계 우선
- **설정 주도**: JSON 구성을 통한 설정 관리

## 프로젝트 구조

```
src/
├── main/           # Electron 메인 프로세스 (Node.js)
│   ├── api/        # API 클라이언트 함수
│   ├── server/     # Hono HTTP 서버
│   ├── window/     # 윈도우 관리
│   └── ipc/        # IPC 핸들러
├── renderer/       # Vue 프론트엔드
│   ├── views/      # 페이지 컴포넌트
│   ├── stores/     # Pinia 상태 관리
│   ├── components/ # 재사용 가능한 UI 컴포넌트
│   └── utils/      # 프론트엔드 유틸리티
├── preload/        # Preload 스크립트 (보안 브리지)
├── types/          # TypeScript 타입 정의
├── zod-schema/     # 런타임 검증 스키마
└── config/         # 애플리케이션 설정
```

## 시작하기 (Getting Started)

### 사전 요구사항
- **Node.js**: 18 이상
- **패키지 매니저**: pnpm (권장)

### 개발 환경 설정
```bash
# 의존성 설치
pnpm install

# 네이티브 모듈(SQLite) 리빌드
pnpm run rebuild

# 개발 서버 시작
pnpm run dev
```

### 빌드 프로세스
```bash
# 프로덕션 빌드
pnpm run build

# 빌드된 애플리케이션 미리보기
pnpm run preview
```
